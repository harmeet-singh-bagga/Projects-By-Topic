## Business Context: What This Data Represents

This problem is set in the context of an app-based gas delivery marketplace. Chama appears to connect customers who want to buy gas cylinders with nearby dealers who can fulfill the order. Like many marketplace or delivery apps, the company needs to decide which dealer option should be shown to the customer, what price should be recommended, and why one option wins over another.

The raw `case.json` file contains backend event data generated by the application. These events capture what happened inside the pricing and offer-selection systems. The goal is to transform this technical event data into structured CSV tables that business, analytics, pricing, and operations teams can use.

At a high level, the data relates to two business processes:

### 1. Dynamic Pricing

Dynamic pricing refers to the system deciding or recommending prices for an offer.

In this problem, dynamic pricing appears in two forms:

* **Price per option**
* **Recommended price range**

The `DynamicPriceOption.csv` file captures option-level prices. Each row tells us the best price calculated for a specific offer option.

Important fields include:

* `Provider`: the pricing algorithm or method used.
* `OfferId`: the customer/order offer being priced.
* `UniqueOptionId`: a specific dealer/product option within that offer.
* `BestPrice`: the price recommended for that option.
* `EnqueuedTimeSP`: when the pricing event happened, converted to São Paulo/Brazil time.

The `DynamicPriceRange.csv` file captures price-range recommendations for an offer.

Important fields include:

* `MinGlobal`: the theoretical/global minimum price.
* `MinRecommended`: the recommended lower price boundary.
* `MaxRecommended`: the recommended upper price boundary.
* `DifferenceMinRecommendMinTheory`: the difference between the theoretical minimum and the recommended minimum.

Business-wise, this helps the company understand how prices are being generated, whether pricing recommendations are within acceptable boundaries, and how pricing changes over time.

### 2. Offer Curation

Offer curation refers to the process of choosing which dealer/product option should be selected, ranked, or shown as the winner for a customer.

The `CuratedOfferOptions.csv` file captures all the candidate options considered for an offer.

Each row represents one available option from a dealer. The system compares these options based on price, availability, estimated delivery time, dealer status, product brand, and internal scoring.

Important fields include:

* `CurationProvider`: the logic used to curate or rank the options, such as by price.
* `OfferId`: the offer/order being evaluated.
* `DealerId`: the dealer providing the option.
* `UniqueOptionId`: the unique identifier for that specific option.
* `OptionId`: another identifier combining dealer/product information.
* `IsMobileDealer`: whether the dealer is mobile.
* `IsOpen`: whether the dealer is currently open.
* `Eta`: estimated delivery time.
* `ChamaScore`: an internal score for the dealer or option.
* `ProductBrand`: the gas brand, such as ULTRAGAZ, CONSIGAZ, LIQUIGAS, or NACIONALGAS.
* `IsWinner`: whether this option won the curation process.
* `MinimumPrice`, `MaximumPrice`, `DynamicPrice`, `FinalPrice`: price-related fields used to evaluate the option.
* `DefeatPrimaryReason`: the main reason an option lost.
* `DefeatReasons`: all reasons why the option lost.

Business-wise, this table helps answer questions like:

* Which dealer option won for each offer?
* Why did certain options lose?
* Were they closed, too expensive, too slow, or already had a driver assigned?
* How do price, ETA, brand, and availability affect the winning option?

So the overall business purpose of this assignment is not just to flatten JSON. It is to convert raw app/backend events into clean analytics tables that can explain pricing decisions and offer-selection decisions.

## Conceptual Approach: Extracting Required Columns from Nested JSON

The main idea in this problem is to convert semi-structured event data into flat, analytics-ready tables. Each row in the original `case.json` file represents one event. The important columns are:

* `EnqueuedTimeUtc`: timestamp of the event
* `EventName`: type of event
* `Payload`: event-specific JSON data stored as a string

The key point is that the required CSV outputs are not directly available as columns in the original file. Most of the useful fields are inside the nested `Payload` column, so the task is really about parsing that payload, identifying its structure, and flattening it into rows.

### 1. First identify the event type

The `EventName` column tells us which kind of payload we are dealing with.

In this dataset, there are two event types:

* `DynamicPrice_Result`
* `CurateOffer_Result`

This is important because both events have different payload structures. So the first step is to split or filter the data by `EventName`.

For example:

* `DynamicPrice_Result` is used to create:

  * `DynamicPriceOption.csv`
  * `DynamicPriceRange.csv`

* `CurateOffer_Result` is used to create:

  * `CuratedOfferOptions.csv`

This is a common pattern in API/event data problems: one raw event table may contain multiple event types, and each event type may need a different flattening logic.

### 2. Parse the `Payload` column

The `Payload` field looks like JSON, but it is stored as a string. So before extracting anything from it, we need to convert it into a Python object using `json.loads()`.

After parsing:

* JSON objects become Python dictionaries.
* JSON arrays become Python lists.
* Nested objects remain nested dictionaries or lists.

Once parsed, we can access values using normal Python indexing, such as:

```python
payload["provider"]
payload["offerId"]
payload["algorithmOutput"]
```

### 3. Understand whether the nested data is a dictionary or a list

This is the most important thing to check in similar problems.

For `DynamicPrice_Result`, the payload contains:

```python
provider
offerId
algorithmOutput
```

But `algorithmOutput` behaves differently depending on the provider.

If the provider is `ApplyDynamicPricePerOption`, then `algorithmOutput` is a list of option-level prices. That means one event can generate multiple output rows.

Conceptually:

```text
One event
  → one provider
  → one offerId
  → many uniqueOptionId values
  → many rows in DynamicPriceOption.csv
```

So we loop through each item inside `algorithmOutput`.

If the provider is `ApplyDynamicPriceRange`, then `algorithmOutput` is a dictionary containing range-level values.

Conceptually:

```text
One event
  → one provider
  → one offerId
  → one algorithmOutput dictionary
  → one row in DynamicPriceRange.csv
```

So we extract the range fields directly.

This is a useful general rule:

```text
If the nested object is a list, loop through it and create one row per item.
If the nested object is a dictionary, extract its keys directly into one row.
```

### 4. Repeat parent-level fields for each child row

When flattening nested JSON, fields from the parent level often need to be repeated for every child row.

For example, in `DynamicPriceOption.csv`, each option has its own:

* `UniqueOptionId`
* `BestPrice`

But the following fields come from the parent event/payload:

* `Provider`
* `OfferId`
* `EnqueuedTimeSP`

Since one offer can contain many options, we repeat the same provider, offer ID, and timestamp for each option row.

This is another common flattening pattern:

```text
Parent fields + child fields = one flat output row
```

### 5. Flatten the curated offer payload using nested loops

The `CurateOffer_Result` payload is more deeply nested.

Its structure is roughly:

```text
Payload
  → list of curated offer/dealer records
      → curationProvider
      → offerId
      → dealerId
      → options
          → list of option records
```

So the extraction requires two loops:

```text
For each event:
    parse Payload

    For each dealer/curation block in Payload:
        extract curationProvider, offerId, dealerId

        For each option inside options:
            extract option-level fields
            combine parent fields + option fields
            create one flat row
```

The final `CuratedOfferOptions.csv` is option-level, not dealer-level. That means each row represents one option, while dealer-level fields like `DealerId` are repeated for every option under that dealer.

### 6. Handle optional fields carefully

Some option records contain:

* `defeatPrimaryReason`
* `defeatReasons`

But winning options may not have these fields.

So we should not directly access them like this:

```python
option["defeatPrimaryReason"]
```

because it may cause an error if the key does not exist.

Instead, use a safe pattern:

```python
option.get("defeatPrimaryReason", "")
option.get("defeatReasons", "")
```

This is important in real API data because not every record has the same fields. Optional fields should be handled with defaults.

### 7. Convert timestamp before writing output

The original timestamp is in UTC. The assignment asks for Brazilian timezone UTC-3 and date format `DD/MM/YYYY`.

So we first parse the timestamp as a datetime, convert it to UTC-3, and then format it as:

```text
day/month/full year
```

For example:

```text
2021-09-05 08:04:08 UTC
```

becomes:

```text
05/09/2021
```

This timestamp transformation should be done once and then reused across the output tables.

### 8. Build rows as dictionaries, then convert to DataFrame

A clean way to solve these problems is to build a list of dictionaries.

Each dictionary represents one output row:

```python
rows.append({
    "Provider": provider,
    "OfferId": offer_id,
    "UniqueOptionId": unique_option_id,
    "BestPrice": best_price,
    "EnqueuedTimeSP": enqueued_time_sp
})
```

After all rows are collected, we convert the list into a DataFrame:

```python
output_df = pd.DataFrame(rows)
```

This approach is simple, readable, and works well for nested JSON because each loop naturally produces one flat row.

### 9. Match the required output schema exactly

After extracting the fields, the final step is to make sure the output DataFrame has:

* the exact required column names
* the exact required column order
* the correct date format
* the required CSV filenames
* the required quoting behavior for string fields

This matters because many data transformation assignments are graded not just on logic, but also on whether the final output format matches the specification exactly.

### General checklist for similar JSON flattening problems

When facing a similar problem, look for these things first:

1. What is the outer structure of the file?
2. Which column contains the actual nested JSON?
3. Is that nested JSON stored as a string?
4. Which column tells me the event type or payload type?
5. Does each event type have a different schema?
6. Inside the payload, which fields are parent-level fields?
7. Which fields are nested child-level fields?
8. Is the nested data a dictionary or a list?
9. Does one input row produce one output row or many output rows?
10. Are any fields optional or missing in some records?
11. Do timestamps need timezone conversion?
12. Does the final CSV require a specific column order, naming convention, or quoting format?

The core skill is not memorizing the code. The core skill is learning to inspect the JSON hierarchy and decide what level each output table represents.
