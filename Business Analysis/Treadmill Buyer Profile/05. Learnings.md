# Gap Analysis Learnings

| Feature / Logic Block | My Draft Approach | Official Solution Approach | Gap / Key Takeaway | Winner & Why |
| :--- | :--- | :--- | :--- | :--- |
| Data ingestion & path handling | Reads `aerofit_treadmill_data.csv` directly with a hard-coded local filename. | Reads from `datasets/aerofit_treadmill_data.csv`, also hard-coded. | Both approaches are environment-dependent. A reusable notebook should detect common file locations and fail with a clear message. | Hybrid — robust path discovery is better than either hard-coded path. |
| Schema, nulls, and duplicate checks | Starts analysis immediately after loading data. | Checks shape, columns, dtypes, nulls, duplicates, and converts string columns to categorical dtype. | The draft misses basic data-quality gates. These checks matter even for small EDA assignments because downstream percentages can be wrong if nulls/duplicates exist. | Official — more complete validation and schema hygiene. |
| Statistical summary | Uses `describe()` only for Income and Miles before binning. | Produces numeric summary, skewness, value counts, and unique-value checks. | The draft correctly inspects variables before binning, but does not summarize the full dataset. | Official — broader baseline profiling. |
| Product-wise contingency tables | Uses concise `pd.crosstab(..., normalize='index')` for product-wise distributions. | Uses crosstabs with margins plus several manually calculated probabilities. | Draft is cleaner for `P(feature \| Product)`, while official better recognizes that the assignment needs marginal and conditional probabilities. | Hybrid — keep draft’s concise crosstab style, extend it into a reusable probability-table function. |
| Conditional probability direction | Mostly computes `P(segment \| Product)` because rows are normalized by Product. | Computes some `P(Product \| segment)`, `P(segment \| Product)`, and marginal probabilities. | Both probability directions are useful but answer different business questions. The final solution should explicitly label row-wise, column-wise, joint, and marginal probabilities. | Official logic, Draft implementation style — official covers more probability types; draft is more idiomatic. |
| Customer segmentation / binning | Creates useful custom Age, Income, and Miles groups. | Creates broader Age and Income categories; does not use a reusable segmentation function. | Draft’s bins are intuitive and business-friendly, but the Income bin using `right=False` can exclude the exact maximum value unless guarded. | Draft, with fixes — preserve the idea but use safe bins with `np.inf` and `include_lowest`. |
| Visual EDA | No plots in the submitted draft. | Covers univariate, bivariate, multivariate, correlation, and outlier plots. | The assignment asks for tables and charts; the draft’s table-first approach is useful but incomplete. | Official — necessary for a complete descriptive analytics submission. |
| Correlation analysis | Not included. | Includes covariance/correlation and a heatmap. | Correlation helps explain why Fitness, Usage, and Miles jointly define the premium KP781 profile. | Official — important analytical layer. |
| Outlier handling | Not included. | Detects outliers using boxplots/IQR, briefly removes them, then concludes they should not be removed. | The official conclusion is right: high Income/Miles values are business signal for KP781, not noise. The final solution should quantify outliers but retain them. | Official principle, Hybrid execution — quantify and retain rather than permanently filtering. |
| Business recommendations | Provides a concise customer-base summary, especially for KP781. | Provides broader actionable insights and recommendations. | Draft’s final profile is intuitive, but needs more quantified evidence and product-by-product recommendations. | Hybrid — combine draft’s direct profile framing with official’s breadth. |
| Code quality and maintainability | Very concise, but notebook is mostly a sequence of ad hoc cells and markdown notes. | More comprehensive, but repetitive manual code and a few non-idiomatic patterns such as string checks with `is not`. | The final solution should be modular: reusable plotting, segmentation, crosstab, and outlier functions. | Hybrid — draft is simpler; official is broader; final should be modular and labeled. |
| Performance and efficiency | Uses vectorized pandas crosstabs; efficient for this dataset. | Also mostly vectorized, but repeats similar operations manually. | For 180 rows, performance is not a bottleneck; maintainability matters more. A helper-based pandas workflow avoids repeated logic and reduces error risk. | Draft for simplicity, Hybrid for final reusable structure. |
