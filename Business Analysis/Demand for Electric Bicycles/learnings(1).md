| Data Ingestion & Schema | Loaded `bike_sharing.csv`, displayed the full DataFrame, and converted `datetime` to datetime. | Loaded the dataset, printed shape, date range, `head()`, `info()`, and descriptive statistics. | Draft started correctly but did not validate schema, row counts, time coverage, or data types beyond `datetime`. | Official, because it establishes dataset integrity before analysis. |

| Visualization Logic | Used crosstabs of categorical variables against `casual` and boxplots grouped by `casual`; labels still referenced product-style analysis. | Used demand-focused plots: year/month/hour/day trends, season/weather boxes, hour-by-workingday interaction, temperature scatter. | `casual` is a component of the target, not a grouping variable for demand drivers; the draft plots are not aligned with the business question. | Official, because the visuals directly explain demand. |
| Outlier Handling | Implemented a useful reusable IQR function and outlier summary, but grouped outliers by `casual`, which is not meaningful here. | Did not emphasize a separate IQR audit. | IQR audit is useful for transparency, but demand peaks should not be blindly removed because they are business-critical. | Draft, with correction: keep the IQR audit, but do not use `casual` as grouping or remove high-demand peaks automatically. |

| Effect Size Reporting | No effect sizes. | Reported epsilon-squared for Kruskal-Wallis and rank-biserial correlation for binary comparisons. | Large samples can make tiny effects statistically significant, so effect sizes are needed for business interpretation. | Official, because it separates statistical significance from practical impact. |
| Post-hoc Comparisons | No post-hoc comparisons. | Used Dunn-style post-hoc logic for pairwise season/weather differences. | Knowing that a variable is significant is less actionable than knowing which levels differ. | Official, but implemented in the final notebook with dependency-light Holm-adjusted pairwise Mann-Whitney tests. |
| Continuous Relationships | Correlated all numeric fields, including `casual` and `registered`. | Correlated predictive features and explicitly flagged `casual`/`registered` as target leakage. | `count = casual + registered`, so using these as predictors would leak the answer. | Official, because it prevents leakage. |
| Multicollinearity | Did not address multicollinearity. | Detected very high `temp`-`atemp` correlation and excluded `atemp` from modeling. | Including both can destabilize linear coefficients and add redundant model inputs. | Official, because it avoids redundant predictors. |
| Temporal Analysis | No hour/month/year trend analysis. | Analyzed year-over-year growth, monthly trends, hourly peaks, and weekday/weekend patterns. | Demand is strongly time-dependent; temporal analysis is central to operations. | Official, because it surfaces the strongest business patterns. |

| Train/Test Split | No train/test split. | Used chronological 80/20 split to avoid future-data leakage. | Random splits would overstate forecasting performance on time-ordered data. | Official, because chronological validation is realistic. |
| Baseline Comparison | No baseline. | Compared against global mean, hour mean, and hour-by-weekday mean. | Model performance needs context; a model should beat simple scheduling rules. | Official, because it establishes lift over simple operational heuristics. |
| Residual Diagnostics | No diagnostics. | Checked residual autocorrelation using Durbin-Watson and Ljung-Box tests. | Residual autocorrelation indicates remaining time-series structure and suggests possible lag features. | Official, because it identifies model limitations. |

| Final Business Answer | No final business recommendations. | Provided ranked demand drivers, model performance, and operational recommendations. | Stakeholders need conclusions, not only charts. | Official, because it closes the loop from analysis to decisions. |

## Final Integration Strategy

The final notebook preserves the draft's concise EDA and reusable IQR-audit idea, but replaces the incorrect `casual`-grouped visuals with demand-centered temporal, statistical, and modeling analysis from the official solution. It also improves the official solution by using modular helper functions, a robust data-path loader, schema validation, dependency-light Holm-adjusted pairwise tests, sklearn pipelines, leakage controls, chronological validation, baseline comparison, and aggregated Random Forest feature importance.
